# Digital Time Machine

Клієнтський SPA у вигляді газети-хроніки: користувач вводить довільну дату, застосунок збирає для неї курси валют, топ-пісні, фільми, історичні події, меми та відеотренд і рендерить усе як розворот часопису. Власного бекенду немає, дані отримуються напряму з браузера з трьох зовнішніх безключевих API та з локальної кураторської бази даних для категорій, де стабільного публічного API з історичними вибірками немає.

### Шари застосунку

**Presentation** — `Portal.jsx`, `Dashboard.jsx`, `PageStains.jsx`, `NewspaperArticle.jsx`. `Portal` містить власний компонент `Typewriter` (посимвольна анімація тексту через `setTimeout`) і валідацію введеної дати (день 1–31, рік 1900–2026) до виклику `onLaunch`. `Dashboard` — переважно презентаційний: отримує вже завантажені дані через props, не звертається до `api.js` напряму, але сам керує Spotify-подібним станом відтворення пісень і YouTube-плеєра (`useRef`/`useEffect` для ініціалізації та знищення плеєра).

**Business logic / orchestration** — `App.jsx` (113 рядків). Тримає стан `selectedDate`, `loading`, `timeMachineData`, `error`, перемикає екрани через `AnimatePresence` з `mode="wait"`. Єдина точка виклику завантаження даних — `handleLaunch`, який делегує все `fetchAllTimeMachineData`.

**Data access** — `services/api.js` (457 рядків). Найбільший і найважливіший файл проєкту: шість функцій отримання даних (`fetchRates`, `fetchSongs`, `fetchMovies`, `fetchNews`, `fetchMemes`, `fetchYoutube`) і одна кураторська база даних (фільми по роках з 1970 по 2024, меми по роках з 1998 по 2024, знакові відео YouTube по роках з 2005 по 2023). Локального зберігання (localStorage) у проєкті немає — увесь стан живе лише в пам'яті на час сесії.

### Стратегія отримання даних

Категорії поділяються на дві групи за принципово різним підходом:

- **Живі дані з API, з фолбеком**: курси валют (Frankfurter, з математичним фолбеком через `Math.sin`/`Math.cos` від року при недоступності API), пісні (iTunes Search, з фолбеком на три захардкоджені треки), історичні події (Wikimedia "On This Day" — вибірка сортується за близькістю року події до обраного, беруться 4 найближчі; фолбек — одне узагальнене повідомлення).
- **Кураторські локальні дані, без звернення до стороннього API історичних вибірок**: фільми, меми, YouTube-тренди. Для фільмів текстовий опис береться з локальної бази, а обкладинка — з живого запиту до iTunes Search за назвою фільму (`fetchMovies` робить `Promise.all` із запитом на кожен тайтл, з `encodeURIComponent` для назви), із фолбеком на попередньо прописаний URL зображення Unsplash.

Функція `getCuratedItem` реалізує пошук найближчого попереднього року в кураторській базі: якщо точного року немає, береться найближчий менший рік з ключів об'єкта.

### Потік даних

```
Portal → onLaunch(dateString)
  → App.handleLaunch → fetchAllTimeMachineData(dateString)
      ├── Promise.all: fetchRates, fetchSongs(year), fetchNews(dateString, year)
      ├── fetchMovies(year)       — послідовно після Promise.all, сам робить внутрішній Promise.all по обкладинках
      ├── fetchMemes(year)        — синхронний виклик, без мережі
      └── fetchYoutube(year)      — синхронний виклик, без мережі
  → setTimeMachineData(data) → рендер Dashboard
```

### Процедурна графіка: плями на сторінці

`PageStains.jsx` — генератор 1–3 "плям" (кавові кільця, чорнильні ляпки, розмазані сліди), унікальних для кожної обраної дати. Ключова деталь: генерація детермінована — власний seeded PRNG (mulberry32, ініціалізований хешем від рядка дати) означає, що для однієї й тієї ж дати плями завжди виглядають однаково при повторному відкритті, а не рандомізуються на кожен рендер. Позиції обираються з шести фіксованих зон по краях сторінки (щоб не перекривати контент), форма плям спотворюється через SVG `feDisplacementMap`/`feTurbulence` для органічного, неправильного контуру замість ідеальних геометричних фігур. Шар суто декоративний: `position: fixed`, `pointer-events: none`, `z-index: 0`.

### Дизайн

Дизайн-токени в `index.css`: кремовий фон `#f7f4eb`, вугільний текст `#1a1a1a`, акцентний червоний `#9e1a1a`, три шрифтові стеки (serif — Playfair Display, sans — Space Grotesk, mono — Share Tech Mono). Текстура паперу на `body::before` побудована шаруванням шести CSS-фонів: SVG-шум через `feTurbulence` (зернистість), радіальний віньєт, три діагональні `linear-gradient` під різними кутами (імітація згинів), два радіальні градієнти для нерівномірного освітлення — усе з `mix-blend-mode: multiply`, `z-index: -1` і `pointer-events: none`. Поверх цього шару `PageStains` додає ще один рівень фізичності — плями конкретного дня.

### Безпека на рівні документа

`index.html` містить `Content-Security-Policy` meta-тег: `default-src 'self'`, `connect-src` явно перелічує лише три реально використані API-хости (`en.wikipedia.org`, `itunes.apple.com`, `api.frankfurter.app`), `img-src` обмежений `mzstatic.com` (обкладинки iTunes) і `images.unsplash.com` (фолбек-зображення), `frame-src` — лише `youtube.com` для вбудованих відео. Зовнішні посилання (`<a target="_blank">`) мають `rel="noopener noreferrer"`.

## Функціонал

- Введення довільної дати (день/місяць/рік) з валідацією діапазону (рік 1900–2026) і повідомленням про помилку.
- Курси USD до EUR, GBP, JPY, CAD, CHF на обрану дату (Frankfurter, історичні дані від 4 січня 1999 року — до цієї межі дата обрізається автоматично).
- До 4 історичних подій дня, відсортованих за близькістю року до обраного, з посиланням на Wikipedia.
- 3 фільми, актуальні для року (кураторська вибірка з живими обкладинками з iTunes).
- Пісні-хіти року (iTunes Search) з інлайн-прослуховуванням 30-секундного прев'ю; при відтворенні одного трека інші автоматично ставляться на паузу.
- 3 меми, характерні для року (кураторська база 1998–2024).
- Одне знакове відео на YouTube для року, вбудоване як iframe (кураторська база 2005–2023; для дат до 2005 — повідомлення "YouTube ще не існував").
- Унікальні для кожної дати "сліди часу" на сторінці — кавові плями й чорнильні ляпки, що завжди однакові при поверненні до тієї самої дати.
- Анімовані переходи між екранами вибору дати й дашборда (Framer Motion, `AnimatePresence`), stagger-анімація появи рубрик.
- Динамічний номер випуску і поточна дата в шапці, розраховані на льоту.

## Встановлення та запуск

Node.js >= 20.

```
npm install
npm run dev       # Vite dev-сервер, шлях з префіксом /DigitalTimeMachine/
npm run build     # артефакти в ./dist, base '/DigitalTimeMachine/'
npm run lint      # oxlint
```

`base` у `vite.config.js` заданий безумовно як `/DigitalTimeMachine/` — застосовується і до збірки, і до dev-сервера. Зовнішні API (Frankfurter, iTunes Search, Wikimedia REST) не потребують ключів чи реєстрації — застосунок працює одразу після встановлення залежностей.

Деплой автоматизований через GitHub Actions (`.github/workflows/deploy.yml`): тригер — push у `main` або ручний запуск, `npm install` + `npm run build` на Node 20, публікація через `actions/deploy-pages` в оточення `github-pages`.

## Тестування

Тестів немає. Лінт-конфігурація (`.oxlintrc.json`) вмикає `react/rules-of-hooks` як помилку та `react/only-export-components` як попередження; у CI не інтегрована — workflow виконує лише build і deploy.

## Обмеження

Фолбек курсів валют при недоступності Frankfurter генерує значення математичною функцією від року (`Math.sin`/`Math.cos`), а не реальними історичними даними — у разі збою API показані цифри є правдоподібними, але вигаданими, без будь-якого попередження в інтерфейсі про це. `fetchMovies` робить окремий мережевий запит до iTunes на кожен фільм у вибірці — для 3 фільмів це 3 послідовно ініційованих запити без кешування, при повторному відкритті того самого року обкладинки запитуються заново. Кураторські бази фільмів, мемів і відео обмежені конкретними роками (фільми — 1970–2024, меми — 1998–2024, YouTube — 2005–2023); для дат поза цим діапазоном `getCuratedItem` повертає дані найближчого меншого року без позначення в інтерфейсі, що це не точний рік. CSP не задає `frame-ancestors` — сторінку теоретично можна вбудувати в чужий `<iframe>` (захист від clickjacking відсутній).
